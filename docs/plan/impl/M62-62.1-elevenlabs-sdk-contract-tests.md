# M62 Task 62.1: ElevenLabs SDK Contract Tests - Implementation Plan

## Overview

This plan covers the verification layer for
[M62: ElevenLabs Speech-to-Text API Parity](../milestones/M62-elevenlabs-api-parity.md).

The goal is not to replace all existing ElevenLabs tests. The goal is to add the missing
consumer-contract layer: a pinned-version suite that uses the official
[`elevenlabs-python`](https://github.com/elevenlabs/elevenlabs-python) SDK against
Dalston's ElevenLabs-compatible surface.

This document prepares the development work only. It does not imply any implementation
has already been started.

---

## Goal

Make the official ElevenLabs Python SDK part of the required compatibility gate so that
subtle contract drift is caught early:

- request serialization drift
- response parsing drift
- status-code and polling drift
- auth/token contract drift
- realtime client setup drift

The SDK suite should become the primary executable parity contract, while the existing
route-level tests remain useful for narrow validation and error branches.

---

## Non-Goals

- Replacing all current route-level integration tests with SDK tests
- Using live ElevenLabs API calls in CI
- Building a second test harness just for ElevenLabs if an existing Dalston test harness
  can be reused
- Treating the SDK as the source of truth for engine_id semantics it does not expose

When the SDK does not fully specify behavior, the fallback should be frozen raw
HTTP/WebSocket traces captured in M62 Phase 0.

---

## Test Layers

The intended verification stack is:

### 1. Route-Level Integration Tests

Purpose:

- fast validation of narrow request/response branches
- explicit error-path coverage
- deterministic adapter/unit-style assertions

Current anchor:

- [tests/integration/test_elevenlabs_api.py](../../../tests/integration/test_elevenlabs_api.py)

Planned role after M62:

- keep these tests
- narrow them to handler- and adapter-level assertions
- stop using them as the primary public-contract oracle

Important cleanup when development starts:

- remove stale assumptions such as arbitrary `model_id` acceptance
- remove stale assumptions such as `200 {"status":"processing"}` polling responses

### 2. SDK Contract Integration Tests

Purpose:

- assert the official client can talk to Dalston unchanged except for `base_url`
- catch request serialization and parsed-response mismatches
- verify the client-facing contract end to end without a full Docker stack

Planned location:

- `tests/integration/test_elevenlabs_sdk_contract.py`

Harness shape:

- start Dalston in a deterministic test harness
- point `ElevenLabs(base_url=...)` at that harness
- use dependency overrides or a lightweight test server where possible
- keep engine_id fast enough for normal integration execution

### 3. Live-Stack SDK Smoke Tests

Purpose:

- prove the pinned SDK still works against the real local stack
- catch wiring issues that the deterministic harness cannot

Planned location:

- `tests/e2e/test_elevenlabs_sdk.py`

Scope:

- intentionally narrow
- one or two happy-path calls per major surface
- no broad error-matrix coverage here

### 4. Raw Realtime / Protocol Tests

Purpose:

- cover protocol translation details the SDK does not expose cleanly
- keep failure diagnosis fast and local

Likely location:

- existing realtime integration tests or a new ElevenLabs-specific realtime test file

---

## Contract Inputs

The SDK suite should be driven by three locked inputs from M62 Phase 0:

1. Public ElevenLabs STT docs as of **March 8, 2026**
2. A pinned `elevenlabs-python` version
3. Frozen raw HTTP/WebSocket traces for engine_id details the SDK does not encode

These inputs should be checked into the repo so parity work does not drift while the
public docs or the SDK evolve.

## Required SDK Coverage

The pinned integration suite should cover the consumer-visible contract that is easiest
to break accidentally.

### Batch Coverage

- `speech_to_text.convert(...)` sync happy path
- `speech_to_text.convert(...)` async happy path
- `speech_to_text.get(...)`
- `speech_to_text.delete(...)`
- validation failure for unsupported `model_id`
- validation failure for invalid keyterms
- request-location behavior for `enable_logging`
- `additional_formats` where supported
- `entity_detection` and `entities` where supported
- webhook request fields `webhook`, `webhook_id`, `webhook_metadata`

### Auth and Token Coverage

- single-use token issuance on the official ElevenLabs path
- realtime connect via `?token=`
- second use of the same token rejected

If the SDK exposes a typed helper for token creation, use it. If not, rely on the frozen
Phase 0 raw HTTP fixture for the issuance path and keep the SDK suite focused on the
SDK-visible surface.

### Realtime Coverage

- SDK-visible realtime connection setup
- manual commit flow
- session-start payload parsing
- basic final-transcript flow

If the official SDK does not expose a stable helper for part of the realtime surface,
cover that part with raw WebSocket tests driven by the Phase 0 frozen fixtures rather
than inventing a fake SDK wrapper.

## Proposed Test Cases

### Phase 0 Skeleton

- SDK package import and version pin sanity
- basic client construction with `base_url`
- one minimal sync transcription fixture
- fixture loading for frozen SDK/raw traces

### Phase 1 Contract Cases

- sync convert success
- async convert success with populated `request_id`
- GET pending returns the expected client-visible not-ready behavior
- DELETE works through the ElevenLabs alias
- keyterm > 5 words fails with the expected public error shape
- unsupported `model_id` fails explicitly
- realtime `commit_strategy` default is visible as `manual`
- `ulaw_8000` no longer produces client-visible corruption in the supported path

### Phase 2 Contract Cases

- single-use token issuance and consume-once behavior
- `webhook=true`, `webhook_id`, `webhook_metadata`
- `entity_detection` request and `entities` response
- `additional_formats` inline response
- `previous_text` and `include_language_detection`
- exact `words[]` payload including `logprob`, `characters`, and correct `type`

### Phase 3 Smoke Cases

- live-stack SDK sync transcription
- live-stack token + realtime connect smoke

## CI Policy

- Pin the primary compatibility gate to one known `elevenlabs` version
- Run the pinned SDK suite in normal integration CI
- Keep the live-stack SDK test in the `e2e` lane
- Optionally add a non-blocking latest-version SDK canary later

This keeps the compatibility gate deterministic while still giving visibility into
upstream SDK drift.

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| SDK releases drift faster than docs | Pin one `elevenlabs` version and freeze Phase 0 fixtures |
| SDK does not expose every public route cleanly | Use raw HTTP/WebSocket fixtures only for the non-SDK-visible edges |
| The deterministic harness diverges from the live stack | Keep a narrow live-stack SDK smoke path in `tests/e2e/` |
| Existing mock-heavy tests continue asserting stale behavior | Narrow their scope and stop using them as the public contract oracle |
| Realtime SDK helpers are unstable or incomplete | Lock traces first and keep lower-level realtime tests for exact event coverage |

---

## Development Order

1. Pin the SDK version and capture frozen traces.
2. Create the SDK contract test skeleton and fixture layout.
3. Update the existing route-level ElevenLabs tests so they stop asserting stale parity
   behavior.
4. Add Phase 1 SDK contract cases.
5. Add Phase 2 token, webhook, enrichment, and realtime cases.
6. Add the narrow live-stack smoke path last.

---

## Out of Scope

- Writing or landing any ElevenLabs compatibility code in this task
- Treating route-level mock tests as sufficient proof of public compatibility
- Depending on live ElevenLabs network access for routine CI
